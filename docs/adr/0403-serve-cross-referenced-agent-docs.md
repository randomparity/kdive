# 0403 — Serve cross-referenced agent docs and cite them by resource URI

Status: Accepted

- **Date:** 2026-07-21
- **Issue:** #1361 (P1 agent-surface release blockers), parent epic #1360.
- **Relates to:** [ADR-0151](0151-mcp-doc-resources.md) (the flat, code-defined
  doc-resource allowlist this extends) and [ADR-0270](0270-no-adr-refs-in-agent-surface.md)
  (internal ADRs are deliberately not served — their relative links stay internal).

## Context

Operator/agent guide docs are served over MCP as a **flat `resource://` allowlist**
(`DOC_RESOURCES`, ADR-0151): there is no path traversal, so a Markdown *relative* link
(an `errors.md` sibling reference) in a served doc cannot be followed by an agent — the
client has only the `resource://kdive/...` URIs advertised in `list_resources`. Three served docs
carried relative links to targets an agent could not reach:

- `agent-index.md` → `../operating/external-build-upload.md` — the target **is** served,
  but the relative link form is unfollowable.
- `response-envelope.md` → `errors.md`, `safety-and-rbac.md`, `async-jobs.md` — none were
  served at all.
- `toolsets/control.md` → `../../operating/race-debugging.md` — not served.

These are all core agent-facing guides (how to read errors, the RBAC/redaction contract,
the async-job polling contract, the race-reproduction workflow behind
`control.watch_for_crash`) that the citing docs point an agent to mid-investigation.

## Decision

**Serve the four missing targets in the `DOC_RESOURCES` allowlist (audience `all`,
no provider gate), and rewrite every flagged relative cross-doc link to cite the
target's `resource://kdive/...` URI instead of a relative Markdown path.**

This keeps the existing model intact: cross-references an agent is meant to follow use
`resource://` URIs (the idiom `agent-index.md`'s toolset table already uses); relative
Markdown links remain only for **internal** targets an agent is deliberately not given
(ADR-0270 ADRs) and for human-only design/runbook docs that are not served. The rewrite is
applied to **every** relative `*.md` link in a served doc whose target is *itself* served —
not only the three findings F1 named — because serving a doc that dead-links to other served
content reproduces the same defect one hop in. Two regression guards enforce this: one asserts
every `resource://kdive/docs/...` URI in a served doc resolves to an allowlisted resource; the
other asserts no served doc links to another *served* doc by relative Markdown path.

Relative `ADR-NNNN` links (pointing into `../adr/`) inside the newly-served bodies are
unchanged: ADRs stay unserved by design, and `test_no_adr_leak` guards resource *metadata*
(title, description, URI), not body text — already the case for `response-envelope.md`, which has
long been served with ADR links in its body.

## Consequences

- Four new served resources; snapshots regenerated by `just resources-docs` and drift-guarded
  as usual. No provider gate, so they list on every deployment.
- The flagged links **and** every relative link between served docs become `resource://`
  citations — followable over MCP, still readable on GitHub as inline URIs. This touched the
  four newly-served docs and `toolsets/images.md` (a pre-existing instance the new serving
  surfaced).
- Two new tests close the class of bug this ADR fixes: one fails if a served doc cites a
  `resource://kdive/docs/...` URI absent from `DOC_RESOURCES`; the other fails if a served doc
  links to another served doc by relative Markdown path.

## Considered & rejected

- **De-link the dangling references.** Rejected: it strips genuinely useful navigation an
  agent needs mid-investigation (error taxonomy, RBAC contract, async polling, race
  workflow). Serving is consistent with the flat-allowlist model and the existing
  `external-build-upload.md` precedent (an operating doc already served `all`).
- **Inline the referenced content into the citing docs.** Rejected: duplicates hundreds of
  lines across docs and creates drift; the allowlist exists precisely so a cited doc is
  reachable once, not copied.
