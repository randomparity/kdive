# fadump opt-in with a QEMU feature gate on pseries (#1151)

Date: 2026-07-14
Status: approved (design)
Issue: #1151 · Epic: #1139 (full ppc64le support) · ADR: `docs/adr/0349-ppc64le-fadump-opt-in.md`
Depends on: #1148 (kdump on ppc64le — per-arch crashkernel defaults + capture proof, ADR-0346, merged PR#1169)

## Problem

kdump is the crash-capture spine on ppc64le (#1148): the guest kexec-boots a capture
kernel that reads `/proc/vmcore` and runs makedumpfile. **fadump** (firmware-assisted
dump) is a POWER-specific alternative the design decision named alongside kdump
(`2026-07-13-ppc64le-full-support.md` §Crash capture, decision 2). Instead of kexec,
fadump asks the platform firmware to preserve memory across a *memory-preserving reboot*:
on crash the guest issues the `ibm,os-term` RTAS call, the platform preserves the
registered memory regions, and the next boot exposes them as `/proc/vmcore` for the same
kdump userspace scripts to save. It is more reliable than kexec-kdump because the dump is
taken from a fully firmware-reset system.

Three facts make this a gated feature rather than a drop-in:

1. **It needs a firmware RTAS call the emulator may not implement.** fadump requires the
   platform to export `ibm,configure-kernel-dump` (PAPR §7.4.9). QEMU's `pseries` machine
   implements it **only in recent versions** — the RTAS call and its device-tree
   properties (`ibm,configure-kernel-dump{,-sizes,-version}`) landed in **QEMU 10.2**
   (`hw/ppc/spapr_fadump.c`; the series was still in review on qemu-devel through
   Oct 2025 and shipped in the 10.2 release). A fadump-requesting provision on an older
   QEMU must fail **at admission with a clear category**, never boot a guest that hangs
   or silently never captures.

2. **Whether it works end-to-end under TCG is unverified.** The epic flagged this in
   "Known unverified" (issue 12). The memory-preserving reboot is pure `spapr` emulation
   (`trigger_fadump_boot`, `do_fadump_register`), not KVM-HV hardware, so it *should*
   work under TCG the same as under KVM — but that is a hypothesis until a live capture
   proves it. The local host runs **QEMU 10.2.2** (Fedora 44), whose `qemu-system-ppc64`
   binary exports `ibm,configure-kernel-dump`, so a live TCG attempt is warranted (not a
   foregone "requires native POWER" verdict). If the live attempt fails after honest
   iteration, the sub-issue degrades to a documented QEMU floor + POWER10-gated verdict
   without blocking the epic spine (kdump remains the default).

3. **The capture-method vocabulary has no fadump member.** `CaptureMethod`
   (`domain/capture.py`) is `CONSOLE/HOST_DUMP/GDBSTUB/KDUMP`. fadump is a distinct
   method (firmware-assisted, not kexec) but shares kdump's retrieve path, so it must be
   modeled as a first-class method that *reuses* kdump machinery where the mechanism is
   identical — not silently folded into `KDUMP`.

## Inputs (already landed)

- **ADR-0346 / #1148**: the per-arch `crashkernel` default (`arch_traits.default_crashkernel`
  — 512M ppc64le), the host-side `depmod -b` module injection that unblocks a cross-arch
  KDUMP install under the x86_64 libguestfs appliance, and the live-proof scaffold: the
  kdump-enabled `fedora-kdive-ready-44-ppc64le` rootfs, the uploaded ADR-0343 bundle at
  `/home/dave/kdive-ppc-proof`, and the `live_stack` driver
  `test_ppc64le_kdump_captures_a_vmcore_under_tcg`.
- **ADR-0300 / #989**: the per-install `crashkernel` reservation seam, threaded
  `runs.install` → `cmdline_for` → `system_required_cmdline`.
- **ADR-0338/0339/0340 / #1140–1142**: discovery advertises `guest_arches`
  (`{arch: {accel, emulator}}`) on the Resource capabilities JSON column with a defensive

- **ADR-0318 / #1052**: `crash_capture_refusal` gates a kdump vmcore on the uploaded
  kernel config carrying the crash symbols.

## Design

fadump is a **new `CaptureMethod.FADUMP`** that reuses the KDUMP install and retrieve
machinery everywhere the mechanism is identical, and diverges in exactly two places: it
appends `fadump=on` to the boot cmdline, and it is gated at admission on a discovered host
QEMU-feature signal. The alternative (a boolean modifier on the KDUMP path, so
`capture_method()` stays `KDUMP`) was considered and rejected in ADR-0349: a distinct enum
member makes fadump legible in telemetry/surfacing and forces every `is CaptureMethod.KDUMP`
branch to be an explicit share-or-diverge decision, so no capture site silently ignores
fadump. The cost — auditing every `KDUMP` reference — is paid once here and enumerated in §6.

### 1. Profile opt-in (`fadump=on` + the reservation)

`LibvirtDebugOptions` (`profiles/provisioning.py`) — the `debug` block that already declares
which capture methods a System is provisioned for (`preserve_on_crash`, `gdbstub`) — gains:

```python
fadump: bool = False
```

The reservation reuses the existing `LibvirtProfile.crashkernel` token: fadump reads
`crashkernel=` for its boot-memory reservation (the kernel deprecated `fadump_reserve_mem=`
in favour of `crashkernel=`), so **no new reservation field** is added. A fadump System
therefore sets both `crashkernel` (the reservation, method signal) and `debug.fadump=True`
(the firmware-assist opt-in).

**Parse-time invariant** (a new `model_validator(mode="after")` on `ProvisioningProfile`,
mirroring `_pair_boot_method_with_provider`): when the local-libvirt section sets
`debug.fadump=True`, the profile must have `arch == "ppc64le"` **and** a non-`None`
`crashkernel`. fadump is POWER-specific (the RTAS is pseries-only) and, in kdive's model,
a crash-capture System is defined by its reservation token — a `fadump=on` with no
reservation would resolve to a non-capture method and silently drop the flag. Violations
raise `CONFIGURATION_ERROR` at `ProvisioningProfile.parse`, before any host is consulted.

### 2. Capture-method derivation

`LocalLibvirtProfilePolicy.capture_method` nests the fadump branch under the existing
crashkernel check, so FADUMP is only ever resolved for a reserved System:

```python
if section.crashkernel is not None:
    if section.debug.fadump:
        return CaptureMethod.FADUMP
    return CaptureMethod.KDUMP
...
```

A new `fadump_provisioned(profile) -> bool` policy method (mirroring
`host_dump_provisioned`/`gdbstub_provisioned`) exposes the flag for admission and boot
evidence. `CaptureMethod.FADUMP = "fadump"` is added to `domain/capture.py`.

### 3. Boot cmdline

`system_required_cmdline` (`services/runs/steps.py`) emits the reservation for both kdump
and fadump, and appends `fadump=on` only for fadump:

```python
if method in (CaptureMethod.KDUMP, CaptureMethod.FADUMP):
    tokens.append(f"crashkernel={crashkernel or traits.default_crashkernel}")
    if method is CaptureMethod.FADUMP:
        tokens.append("fadump=on")
elif method is CaptureMethod.GDBSTUB:
    tokens.append(_GDBSTUB_NOKASLR)
```

`_PLATFORM_OWNED_CMDLINE_TOKENS` gains `"fadump="` so a caller `runs.install` cmdline
override cannot inject a conflicting `fadump=` token past the platform-owned guard. The
per-arch default reservation (512M ppc64le, ADR-0346) is reused unchanged — fadump's boot
memory sizing is the same order as kdump's, and an operator wanting more passes it through
the ADR-0300 per-install seam. No new `arch_traits` field.

### 4. Discovery: record whether the host QEMU implements pseries fadump

`local_libvirt/discovery.py` `list_resources` records a new capability key alongside
`guest_arches`:

```python
PSERIES_FADUMP_KEY: detect_pseries_fadump(guest_arches),
```

`detect_pseries_fadump(guest_arches) -> bool` reads the ppc64le emulator path already
discovered in `guest_arches["ppc64le"]["emulator"]`, runs `<emulator> --version`, parses
the `QEMU emulator version X.Y.Z` line, and returns `True` iff `(major, minor) >= (10, 2)`
— the **QEMU floor** at which `pseries` exports `ibm,configure-kernel-dump`. Returns
`False` when: no ppc64le guest arch is advertised, the version is below the floor, or the
probe fails (missing binary, unparseable output, non-zero exit). This is **fail-closed** —
uncertainty denies fadump, so a false-positive can never boot a hanging guest; the only
cost of a false-negative (a distro back-porting fadump to an older QEMU) is that fadump is
declined on a host that could support it, which admission reports with an actionable
message. The floor and its fail-closed rationale are recorded in ADR-0349.

`ResourceCapabilities.pseries_fadump() -> bool` (a defensive typed reader mirroring
`guest_arches()`) returns `False` when the key is absent or not a `bool`, so a stale or
hand-edited capabilities row never advertises fadump by accident. `PSERIES_FADUMP_KEY` is
added to `_KNOWN_KEYS`. This rides the existing opaque JSON capabilities column — **no
migration** (the `guest_arches`/ADR-0338 precedent).

### 5. Admission host gate + capability surfacing

- **Admission** (`services/systems/admission.py`): a new
  `_validate_fadump_supported(conn, resource_id, profile)` — a sibling of
  `_resolve_new_system_accel` — is called at the same System-mint point (before the
  granted→active flip). When the profile opts into fadump (`fadump_provisioned`) and the
  bound Resource's `capability_view.pseries_fadump()` is `False`, it raises
  `CONFIGURATION_ERROR` naming the QEMU floor and the host. A `None` resource_id or a
  resource that has never been re-discovered (no `pseries_fadump` key → reader returns
  `False`) is treated as unsupporting — fail-closed. Because the raise happens before the
  flip and is caught by the existing `_failure_from_error` wrap, a rejection **consumes no
  capacity** and returns a typed envelope — never a hang (the issue's core AC).
- **Static method vocabulary** (`local_libvirt/composition.py`): `CaptureMethod.FADUMP` is
  added to the provider's `ProviderSupport.capture_methods` set so `vmcore.fetch` can
  resolve an omitted method through the profile (`_resolve_capture_method` requires the
  method be in both `_VMCORE_METHODS` and the support set) and so `supported_capture_methods`
  advertises it. The provider *can* do fadump; whether a *given host* can is the discovered
  signal + admission gate above.
- **Method vocabulary surfacing** (`mcp/tools/catalog/resources.py`): `_VMCORE_METHODS` gains
  FADUMP (it produces a core), and `supported_capture_methods` lists `"fadump"` automatically
  from the runtime support set. The **per-host** `pseries_fadump` signal is **not** surfaced in
  `resources.get`: `_project_capabilities` receives only the provider descriptor, not the
  Resource row, so surfacing the per-host flag there would thread resource state into a
  descriptor-only projection. Host-specific availability is exposed via the doctor check (§7)
  and enforced at admission — an agent learns a host cannot fadump from the doctor signal or a
  fast admission rejection, not from the resource description.

### 6. Every `CaptureMethod.KDUMP` site — the share/diverge audit (the enum-member safety)

Enumerated from `rg CaptureMethod.KDUMP src/kdive/`. **local-libvirt (in scope):**

| Site | Decision |
|------|----------|
| `profile_policy.py:capture_method` | **diverge** — add FADUMP branch (§2) |
| `services/runs/steps.py:system_required_cmdline` | **diverge** — FADUMP adds `fadump=on` (§3) |
| `local_libvirt/lifecycle/install.py:283` (kdump env absent) | **share** — `in (KDUMP, FADUMP)`; fadump needs the same kdump userspace/modules |
| `local_libvirt/lifecycle/install.py:339` (needs_modules) | **share** — `in (KDUMP, FADUMP)`; fadump's second kernel needs modules |
| `mcp/tools/lifecycle/vmcore/handlers.py:55` (`_VMCORE_METHODS`) | **add** FADUMP — produces a core |
| `mcp/tools/lifecycle/vmcore/handlers.py:255` (ADR-0318 gate) | **share** — `in (KDUMP, FADUMP)`; a fadump core is guest-kernel-produced, needs the crash symbols. The gate stays **kdump-symbol-only** — it does **not** additionally require `CONFIG_FA_DUMP` for FADUMP (§8a records why: the config gate cannot distinguish a fadump kernel from one that silently kdump-falls-back, so the fadump-active *runtime* signal in §8 is the real safeguard, not a static config check) |
| `mcp/tools/lifecycle/runs/steps.py:142` (`_reject_crashkernel_on_non_kdump`) | **share** — `not in (KDUMP, FADUMP)`; a fadump System accepts a crashkernel override |
| `jobs/handlers/runs/install.py:137` (crashkernel backstop) | **share** — `not in (KDUMP, FADUMP)` |
| `jobs/handlers/runs/boot_evidence.py:243` (`inert_capture`) | **share** — append the resolved method value for `KDUMP`/`FADUMP` |
| `local_libvirt/composition.py:158` (support set) | **add** FADUMP (§5) |

| `providers/local_libvirt/retrieve.py:capture` | **unchanged** — dispatch is `if HOST_DUMP … else <overlay harvest>`; FADUMP falls through to the identical `/var/crash/*/vmcore` harvest and is stored as `vmcore-fadump`. **This is why retrieve is shared** |
| `observability/labels.py`, `console/capture_telemetry.py` | unchanged — `capture_method` label takes `method.value` (`"fadump"`) automatically |

**remote-libvirt, fault-inject (out of scope):** `remote_libvirt/*`, `fault_inject/*` KDUMP
sites are untouched — fadump is not offered there (this issue is `provider:local-libvirt`;
remote arch work is a separate epic). Their capture-method sets do **not** gain FADUMP.

### 7. Doctor signal

A `PSERIES_FADUMP_ID` diagnostic check (`diagnostics/checks.py` + a probe mirroring
`MultiarchGdbCheck`/`multiarch_gdb.py`, worker vantage, registered via
`providers/assembly/diagnostics.py`) reports whether the host QEMU implements pseries
fadump, reusing `detect_pseries_fadump`'s version-floor probe. Outcome:
`PASS` (supported), `FAIL`/`MISSING_DEPENDENCY` with a fix hint (upgrade QEMU to ≥10.2 or
that fadump is native-POWER only on this host) when the ppc64le emulator is below the floor,
`ERROR`/undeterminable when no ppc64le emulator is present. The `check-local-libvirt.sh`
preflight gains a matching advisory line.

### 8. Live proof (attempt first; documented verdict fallback)

Reusing the #1148 scaffold, a `live_stack` driver
`test_ppc64le_fadump_captures_a_vmcore_under_tcg` provisions a ppc64le guest under TCG on
the x86_64 host with a **fadump profile** (`crashkernel` reservation + `debug.fadump=True`),
installs the kdump-enabled bundle, `control.force_crash`es it, and harvests the vmcore
through the shared retrieve pipeline.

**Confirm-first preconditions** (named, established before the capture — mirroring #1148's
precondition discipline, not assumed):

- **`CONFIG_FA_DUMP=y` in the baseline/uploaded kernel.** fadump needs it in the *running*
  kernel — a strictly stronger requirement than kdump's `CONFIG_CRASH_DUMP`. The proof
  greps the Fedora ppc64le kernel config (`/boot/config-<ver>` or the packaged config) and
  records it set **before** the capture run. If it is unset, `fadump=on` is silently
  ignored and the kernel kdump-falls-back — so this is a rootfs/kernel gap to close first,
  not an indeterminate capture verdict.
- **kdump userspace + ≥2 GB guest RAM** — the #1148 preconditions fadump reuses (the
  fadump-aware `kdumpctl`, `kexec-tools`, dracut kdump module; the reservation is honored
  only above the distro memory threshold). Recorded as met.

**8a. Discriminating attribution — the mechanism, not just the outcome.** The kernel falls
back to kexec-kdump when fadump cannot reserve or register memory, and still writes a
`/var/crash/vmcore` that kdive stores under `vmcore-fadump` **because the profile said
fadump** — so neither `fadump=on` in the cmdline nor the `vmcore-fadump` key proves fadump
*ran*; both only prove the flag was *set*. The proof therefore records a fadump-specific
**runtime** signal read from inside the guest, which is the authoritative success factor:

- fadump `enabled == 1` **and** `registered == 1` on the provisioned (pre-crash) guest —
  fadump is active and has registered its dump regions with the platform. The proof reads
  whichever sysfs interface the guest kernel exposes: the **modern** directory
  (`/sys/kernel/fadump/enabled`, `/sys/kernel/fadump/registered`) or the **legacy** flat files
  (`/sys/kernel/fadump_enabled`, `/sys/kernel/fadump_registered`) — the path moved across
  kernel versions, so the proof probes the directory first and falls back to the flat files
  rather than assuming one; and
- the guest dmesg `Firmware-assisted dump` activation/registration line (corroborating), and
- the **absence** of a kexec crash-kernel load (`/sys/kernel/kexec_crash_loaded == 0`) — the
  corroborating negative that rules out the kdump fallback path (`registered == 1` is the
  primary positive signal; `kexec_crash_loaded == 1` alongside it would be a misconfiguration
  to record, not necessarily a fallback).

Plus the outcome attribution: the running domain's `<cmdline>` carries `fadump=on`, the
guest `/proc/cmdline` shows `crashkernel=512M fadump=on`, and the retrieved core is a
non-empty `EM_PPC64` vmcore under the `vmcore-fadump` key with makedumpfile fields recorded.

**Pre-registered signals** (named before the run): fadump success = a non-empty `EM_PPC64`
`vmcore-fadump` artifact **and** `fadump_registered==1`/`kexec_crash_loaded==0` observed on
the guest before the crash. A failure records the pre-registered mode — QEMU below floor
(should be caught at admission, not here); `CONFIG_FA_DUMP` unset (caught by the confirm-first
precondition); fadump never registers (`fadump_registered` stays 0 → the kernel kdump-falls-back,
which the runtime signal exposes rather than mislabeling as a fadump success); or the
domain-settled poll times out because fadump reboots to production rather than powering off
(the harvest force-offs and reads the overlay anyway — record whether the core was still
written). Unlike the kdump AC, fadump-under-TCG **may legitimately prove unusable**: if the
capture cannot be made to work after honest iteration, the verdict is **documented** (the
QEMU 10.2 floor recorded in ADR-0349 + "live fadump validation deferred to native POWER,
gated on #1152 POWER10 bring-up") — this is the issue's explicit feasibility-gate fallback,
not shipped as "indeterminate". Either outcome is written to
`docs/design/2026-07-14-ppc64le-fadump-proof-record-1151.md`.

## Acceptance criteria

1. **Profile opt-in renders.** A local-libvirt profile with `debug.fadump=True` +
   `arch=ppc64le` + a `crashkernel` reservation parses; `capture_method` resolves to
   `FADUMP`; `system_required_cmdline` emits `crashkernel=<size> fadump=on` (reservation
   then flag, `fadump=on` last). `debug.fadump=True` with a non-ppc64le arch or with no
   `crashkernel` is rejected `CONFIGURATION_ERROR` at parse. Arch-parameterized unit tests.
2. **Admission rejects on an unsupporting host.** A fadump-opted profile provisioned
   against a Resource whose `pseries_fadump` capability is `False`/absent fails
   `CONFIGURATION_ERROR` at System mint, before the granted→active flip, consuming no
   capacity and returning a typed envelope (never a hang). A Resource advertising
   `pseries_fadump=True` admits it. Unit-tested with a faked capability_view.
3. **Discovery + doctor signal.** `detect_pseries_fadump` returns `True` for a
   `guest_arches` whose ppc64le emulator reports QEMU ≥10.2 and `False` for <10.2 / no
   ppc64le / probe failure; the value is recorded on the Resource capabilities and read
   back via `pseries_fadump()`. The doctor check reports the signal. Unit-tested with the
   version-probe subprocess seam faked.
4. **Retrieve is shared.** A `FADUMP` capture is harvested by the same local retriever
   path as kdump (no `retrieve.py` change) and stored under `vmcore-fadump`; `vmcore.fetch`
   resolves FADUMP for a fadump System's crashed Run. Unit-tested.
5. **Live capture or documented verdict (feasibility gate).** Either a documented
   `live_stack` TCG fadump capture that proves the **mechanism, not just the outcome** — on
   a guest with `CONFIG_FA_DUMP=y` confirmed first, the guest observed pre-crash with fadump
   `registered==1` (via whichever sysfs interface the kernel exposes) and
   `kexec_crash_loaded==0` (ruling out a silent kdump fallback), then a non-empty `EM_PPC64`
   `vmcore-fadump` with `fadump=on` in the domain cmdline and makedumpfile fields recorded —
   **or** a documented verdict that fadump
   requires native-POWER validation, with the QEMU 10.2 floor recorded in ADR-0349 and the
   POWER10 gating named. Written to the proof record.

## Scope / non-goals

- **No new migration / persistence.** The host signal rides the JSON capabilities column
  (ADR-0338 precedent); the opt-in rides the stored profile. `accel` (mig0067) is already
  persisted and unaffected. fadump-on is **not** persisted on the System row — it is
  re-derived from the profile at each read (the profile is the record).
- **No remote-libvirt / fault-inject fadump.** Their capture-method sets and KDUMP sites
  are untouched (separate provider epic).
- **No new reservation field / arch_traits field.** fadump reuses the `crashkernel` token
  and the ADR-0346 per-arch default.
- **No POWER10-native validation.** That is #1152 (gated on hardware); this issue proves
  under TCG or documents the native-only verdict.
- **No `live_vm_tcg` marker** (epic issue 15) and no big-endian ppc64 (epic non-goal).
- **No fadump-specific dracut/rootfs tooling.** The kdump-enabled rootfs userspace (which
  fadump reuses) is the #1148 scaffold; a rootfs that supports kdump supports fadump save.
