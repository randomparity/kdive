# Spec — out-of-band crash-signature console watch (#984)

- **Status:** Draft (for adversarial review)
- **Issue:** #984 — "Add stress/repeat-until-crash-signal reproduction primitive"
- **ADR:** [ADR-0367](../adr/0367-out-of-band-crash-watch.md)
- **Epic:** #1199 (v0.3.0 release-readiness), Workstream D — race diagnosability.

## Problem

Race-condition kernel bugs are found by running a reproducer many times until the timing
window hits and the kernel crashes. When it crashes, **the panic drops the guest's SSH
channel**: whatever the agent was watching over SSH (a `trace_pipe` tail, a drgn poll, the
reproducer's own stdout) dies with the session, so the agent cannot observe the crash it just
provoked from inside the guest. The durable record of the crash is the **serial console**,
which is out-of-band and survives the panic.

Today an agent can drive the reproducer loop over root SSH (the "guest is yours as root"
contract, [ADR-0366](../adr/0366-race-debugging-out-of-band.md)), but there is **no primitive
that watches the out-of-band console for the crash signature while that loop runs** and reports
the first hit deterministically. The only crash-signature match on the console runs during
*boot readiness* (`_CRASH_SIGNATURE` in
`providers/local_libvirt/lifecycle/boot/readiness.py`); nothing watches a *running* (post-boot)
System's console. So the agent must manually poll console artifacts and grep them itself.

## Scope decision (why this shape)

Issue #984 was filed proposing "a job that **iterates a command** while tailing the console,"
scoped to compose "audited in-guest execution (#909) + the readiness crash matcher." That
premise is now stale:

- **#909 (bounded in-guest command execution) was closed** as SSH-equivalent, under the epic's
  governing principle: *a capability earns an MCP tool only if it is out-of-band* (works when
  SSH cannot reach the guest).
- **ADR-0366** then resolved the sibling race-observation issue #986 as documentation and
  wrote that principle down, **explicitly assigning the reproducer loop to "the agent's own
  code run over root SSH"** (ADR-0366 §Context) and naming #984 as the separately-tracked
  primitive.

#984 is therefore a **hybrid**: the *command-loop* half is SSH-equivalent (the agent already
does it), but the *crash-signature-on-the-console* half is genuinely out-of-band — it is the
one thing SSH structurally cannot do, because the crash kills SSH. **This spec builds only the
out-of-band half.** kdive does **not** run agent-supplied commands in the guest; re-introducing
that is exactly the closed-#909 capability the principle forbids.

The two rejected shapes (a full in-guest loop-runner; a docs-only resolution) are recorded in
ADR-0367's alternatives.

### How the acceptance criteria map onto this shape

The issue's acceptance criteria were written for the loop-runner shape. Under the out-of-band
shape they map as:

| Issue acceptance | This spec |
|---|---|
| "iteration count at first signal" | The **agent** owns its loop, so it owns the iteration count. kdive returns **elapsed-to-signal** (`elapsed_s`) — the out-of-band datum it can actually observe — plus the matching console slice. |
| "the matching console slice" | Returned: the redacted console line that matched, with bounded surrounding context. |
| "honours the iteration bound **and** the wall-clock budget" | The **wall-clock budget** (`deadline_s`) is kdive's; the **iteration bound** is the agent's loop. |
| "early-exits deterministically" | The watch returns on the **first** `_CRASH_SIGNATURE` match past the start offset, or at the deadline — no nondeterminism. |

## The primitive

A new durable worker job and its MCP tool.

### Tool: `control.watch_for_crash`

- **Toolset:** `control` — the out-of-band family (`force_crash`, `diagnostic_sysrq`, `power`).
  The watch belongs here because it is the tool for the moment the guest is *about to die and
  SSH is gone*: it reads the console, not the guest.
- **Params:**
  - `system_id: str` — the READY local-libvirt System to watch.
  - `deadline_s: float` — how long to watch before returning "no crash." Defaulted and
    **clamped to a hard maximum** (a watch holds a worker slot for its full duration, so the
    ceiling is a resource bound, not advice). Default `60`, max `300`. A non-finite or
    non-positive value is a `configuration_error`. A value above the cap is clamped to the cap
    (not rejected), matching `jobs.wait`'s `timeout_s` clamp.
  - `idempotency_key: str | None` — replay-safe key, like the other `control.*` tools. Absent,
    every call is a distinct watch (each watch has its own start offset).
- **RBAC:** `contributor`, bound to the target System's project (a leaseholder observing its
  own transient System's console; non-destructive). Same gate as `control.power` /
  `control.diagnostic_sysrq`. `exposure.py` records `_CONTRIBUTOR`.
- **Admission (synchronous, at the tool boundary):** the System exists and is in a granted
  project; the caller has `contributor`; the System is `READY`; the provider is
  `local-libvirt` (the console log is a local-libvirt runtime path). Any failure is a
  `configuration_error` (mirroring `control.diagnostic_sysrq`). On success the tool enqueues a
  `watch_for_crash` job and returns `{job_id, status: queued}`; the agent polls `jobs.wait`.

### Agent workflow

1. `control.watch_for_crash(system_id, deadline_s)` → `{job_id}`. The job snapshots the console
   offset at start, so it only considers output produced **from now on**.
2. The agent drives its **own reproducer loop over root SSH** (compile/`scp` a reproducer,
   `stress-ng`, a fuzzer — per ADR-0366 / `race-debugging.md` Route 3).
3. The agent polls `jobs.wait(job_id)`. When the kernel panics: the console shows the
   signature, the watch job **succeeds** with the crash verdict; the agent's SSH loop dies
   independently, but the out-of-band watch caught it. If the deadline passes with no crash,
   the watch **succeeds** with a "not fired" verdict.

### Job handler behavior (worker)

Modeled on `check_ssh_reachable_handler` (a bounded probe job returning an inline verdict) and
`diagnostic_sysrq`'s lock-free console poll.

1. Load payload; read the System. Re-verify `READY` **at start** (a torn-down System's console
   file may be gone or its loopback reused): not-`READY` → `configuration_error`
   `reason="system_not_ready"`. Re-verify provider is local-libvirt (defense in depth behind
   the tool gate).
2. **Snapshot the start offset** `mark = len(read_console_log(system_id))`. Only
   `console[mark:]` is scanned — a stale boot-time panic already in the retained log cannot
   produce a false hit. (Mirrors `diagnostic_sysrq`'s `mark`.)
3. **Poll loop** (lock-free): every `POLL_INTERVAL_S`, read the console and scan `console[mark:]`
   for `_CRASH_SIGNATURE`. On the **first** match: redact the matched line + bounded context,
   return the **fired** verdict. Continue until `deadline_s` elapses, then take step 5.
   - *Why lock-free is safe:* each poll takes a fresh snapshot and scans a suffix; the local
     serial log only grows while the guest is up (`append="off"` truncates only on power-cycle,
     ADR-0258), so concurrent growth between polls is harmless (a match not seen this poll is
     seen the next). This does **not** assume the guest halts. Several signatures — `detected
     stall` (soft-lockup / RCU stall), and KASAN/KFENCE reports absent `panic_on_warn` — do
     **not** halt the kernel; the watch fires on them anyway (that is desired: catching a
     non-fatal stall is the point), and the console keeps growing after the match without
     affecting the already-returned verdict.
4. **Console truncation guard:** if a later read returns a length `< mark` (the log was
   truncated — a power-cycle happened mid-watch, e.g. `control.power reset`, or a `panic=N`
   auto-reboot), reset `mark = 0` and keep scanning; a truncation means the boot-window content
   is gone, so scanning from 0 is safe and cannot resurface a pre-watch panic. This is logged.
   (A panic that both prints *and* fast-reboots within one `POLL_INTERVAL_S` can still be
   missed on the console; step 5's liveness probe is the backstop that makes the not-fired
   verdict actionable regardless.)
5. **Deadline reached with no match → return `not_fired`.** kdive does not probe guest liveness
   to disambiguate a `not_fired` from a crash that landed outside `console[mark:]` (a panic
   before `mark` from queue latency, or an at-least-once retry that re-snapshots `mark` past an
   already-printed panic; see §The missed-crash window). The agent driving the reproducer over
   SSH already holds the authoritative liveness signal — its SSH drops the instant the kernel
   panics — so a kdive-side probe would be redundant, and a virsh probe would cross the provider
   boundary. The tool and guide document that a `not_fired` paired with a dropped reproducer-SSH
   means "read the full console."
6. **A watch that ran is a success** (fired or not_fired); only an inability to run (System gone
   at start, console unreadable as an infrastructure failure) raises a typed `CategorizedError`.

### Result contract (inline verdict in `result_ref`, the ADR-0164 pattern)

The handler returns a **compact-JSON string** in the job's `result_ref` (exactly as
`check_ssh_reachable` returns its reachability verdict — no new artifact row; the full console
remains durable via the existing console/artifacts tools):

- **Fired:**
  `{"outcome": "fired", "fired": true, "signature": "<matched literal, e.g. 'Kernel panic'>", "matched": "<redacted matched line + bounded context>", "elapsed_s": <float>, "observed_at": "<ISO-8601>"}`
- **Not fired (deadline):**
  `{"outcome": "not_fired", "fired": false, "elapsed_s": <float>, "observed_at": "<ISO-8601>"}`

`matched` is passed through `Redactor(registry=<the worker's global secret_registry>)` — the
same registry `diagnostic_sysrq` uses, injected into the handler via the registrar — so every
secret registered for the worker's lifetime is masked (an empty registry would make redaction a
silent no-op, so the registry wiring is load-bearing, not incidental). `matched` is bounded to
the matched line plus `CONTEXT_LINES` (default 3) lines on each side, and the whole field is
capped at `MATCHED_MAX_BYTES` (default 4096) so the verdict stays compact even if a context line
is pathologically long. `elapsed_s` is the wall-clock from `mark`-snapshot to the returning
poll: on a hit it is `[0, deadline_s + POLL_INTERVAL_S]`; on the deadline path it is
`[deadline_s, deadline_s + POLL_INTERVAL_S]` (one poll interval of slack).

The agent reads the verdict from the job envelope's `refs.result` after `jobs.wait` reports the
job terminal. `suggested_next_actions` on the enqueue envelope is `["jobs.wait"]`.

### The missed-crash window (and why the agent, not kdive, covers it)

`mark` is snapshotted when the **worker** picks the job up, which is strictly after the tool
returns `job_id`. Two windows can put a real panic outside `console[mark:]`, yielding `not_fired`:

- **Queue latency.** The agent gets `job_id` and starts its reproducer. If the panic lands
  before the worker snapshots `mark`, its text is behind `mark`. In practice a race reproducer
  needs many iterations to hit its window, so an iteration-0 crash inside sub-second queue
  latency is unlikely. Agents that want to shrink the window further can confirm the watch is
  `running` (`jobs.get`) before starting the reproducer.
- **Worker-lease loss + retry.** A healthy worker heartbeats its lease on a separate connection,
  so a mid-watch retry happens only on actual worker death; the retry re-snapshots `mark` past
  any panic already printed and returns `not_fired`.

In both windows the agent is the better-positioned observer: it is running the reproducer over
SSH, so its SSH channel drops the instant the kernel panics — a signal kdive does not have and
would only approximate with a probe.

The design does **not** disambiguate these cases with a kdive-side probe. The agent driving the
reproducer over SSH already holds the authoritative liveness signal — its SSH channel drops the
instant the kernel panics. So the contract is: **a `not_fired` verdict paired with a dropped
reproducer-SSH means the crash landed outside the watched window — read the full console** (via
the `artifacts` tools), rather than trusting `not_fired`. The tool docstring and the
race-debugging guide state this. A kdive-side liveness probe (virsh `domstate`) was rejected as
redundant with the agent's own signal and as a provider-boundary crossing (ADR-0367).

### Crash-signature matcher — reuse, single source of truth

The detector is the **existing** boot-readiness matcher `_CRASH_SIGNATURE`
(`providers/local_libvirt/lifecycle/boot/readiness.py`): `Kernel panic`, word-boundaried `BUG:`
/ `Oops:`, `general protection fault`, `unable to handle kernel`, `KASAN:`, `KFENCE:`,
`detected stall` — i.e. exactly the issue's `panic`/`BUG`/`Oops`/`GPF`/`KASAN`/`KFENCE`/
soft-lockup set. The regex + a public helper are **relocated to the domain layer**
(`domain/lifecycle/crash_signatures.py`, beside the existing crash-signature presets) so both
boot readiness and the watch handler import it **boundary-cleanly** — a jobs handler may not
import `providers/local_libvirt` internals (the provider-boundary guard):

```
def first_crash_signature(text: str) -> re.Match[str] | None
```

`readiness.classify_console` imports and calls it (so the regex has one definition); the watch
handler calls it too. `match.group(0)` yields the `signature` field. No behavior change to
readiness.

## Concurrency, safety, observability

- **No advisory lock** is held during the poll (the console read only grows / persists; no
  cross-op exclusion needed — the same property `diagnostic_sysrq` relies on). The watch does
  **not** re-assert `READY` during the poll: the guest *leaving* `READY` by crashing is the
  event being watched for, so requiring `READY` throughout would be wrong.
- **Idempotency / at-least-once:** the job is a read-only observation with no external side
  effect and no persisted row, so a retry is safe to run. A retry re-snapshots `mark` and can
  therefore miss a panic printed during the failed attempt, returning `not_fired`; the agent's
  own dropped-SSH signal covers that case (see §The missed-crash window). The dedup key is
  per-call-unique
  (`{system_id}:watch_for_crash:{idempotency_key-or-uuid4}`), like `power`/`diagnostic_sysrq`.
- **Worker-pool contention.** A watch holds a worker slot for up to `deadline_s` doing nothing
  but polling, so concurrent watches on the shared default lane could delay short lifecycle jobs
  (provision/boot/power). This is bounded, not eliminated, by design choice: the `deadline_s`
  **cap is deliberately modest (max 300s, default 60s)** and the poll yields to the event loop
  between reads (the worker's async loop keeps heartbeating and can interleave). A dedicated
  dispatch lane (`job_dispatch_lane`, mig 0066) is the escalation if contention is ever
  observed — but it is **not** built here: a lane no deployed worker is configured to accept
  would strand every watch job (workers claim only their `accepted_lanes`, default `("default",)`,
  and no production kind uses a non-default lane today), so adding one is a deployment change,
  not a code default. The watch stays on the default lane with a modest cap; the lane option is
  recorded, not implemented (no speculative surface).
- **Cancelable:** `watch_for_crash` joins `CONTRIBUTOR_CANCELABLE_JOB_KINDS` — a contributor
  can cancel its own watch (matching `power`/`diagnostic_sysrq`). Cancellation is cooperative
  via the normal job path.
- **Audit:** the watch is audited like the other control-plane observation jobs (tool,
  `system_id`, `deadline_s`, outcome).
- **Telemetry:** per-kind job telemetry surfaces the watch's success/failure rate, so a
  silently-broken console read (a rising `configuration_error`/`infrastructure_failure` rate
  for `kind=watch_for_crash`) is observable.
- **Redaction:** every console slice that enters the verdict passes
  `Redactor(registry=<worker secret_registry>)` (see the result contract) — the same registry
  and posture as `diagnostic_sysrq`.

## Persistence / migration

`jobs.kind` is `text` with the `jobs_kind_check` constraint (not a PG enum). Adding
`watch_for_crash` requires a forward-only migration `0069_watch_for_crash_job_kind.sql` that
drop-and-recreates `jobs_kind_check` widened with `'watch_for_crash'` (the 0057 pattern; keeps
the constraint name stable for the SQL↔enum tie in `test_migrate.py`). No table or column
change.

## Out of scope (explicit)

- **Running agent-supplied commands in the guest** — that is the closed-#909 SSH-equivalent
  capability; the agent drives its own loop over SSH.
- **A "max iterations" bound** — kdive does not iterate, so there is no iteration bound to
  honour; the agent bounds its own loop.
- **A new artifact/row for the full console** — the full console is already durable via
  `console_rotate` / run console evidence and the `artifacts` tools; the watch only returns the
  matched slice inline.
- **Custom / caller-supplied signature patterns** — no user need is established; the readiness
  matcher is the single source of truth. (Deferred, not designed out.)
- **Non-local-libvirt providers** — the console log is a local-libvirt runtime path; remote
  providers are refused at admission.

## Acceptance criteria (this spec)

1. `control.watch_for_crash(system_id, deadline_s)` on a READY local-libvirt System enqueues a
   `watch_for_crash` job and returns `{job_id, status: queued}`; `jobs.wait` drives it to
   terminal.
2. When a `_CRASH_SIGNATURE` line appears on the console **after** the watch's start offset, the
   job succeeds with `fired: true`, the matched `signature`, a redacted `matched` slice, and
   `elapsed_s`. The match is deterministic (first hit past `mark`).
3. When no signature appears before `deadline_s`, the job succeeds with `fired: false` and
   `elapsed_s ∈ [deadline_s, deadline_s + POLL_INTERVAL_S]`. A pre-watch panic already in the
   retained log does **not** cause a false `fired`.
3a. The verdict is `fired` or `not_fired` only; kdive does not probe guest liveness. A crash that
   landed outside `console[mark:]` (queue-latency or retry window) returns `not_fired` — the agent
   detects it via its own dropped reproducer-SSH and reads the full console (documented on the
   tool and in the race-debugging guide).
4. `deadline_s` is clamped to `[0 < d ≤ MAX]`; a non-finite/non-positive value is a
   `configuration_error`. A non-READY or non-local-libvirt System is a `configuration_error`.
   A caller without `contributor` on the project is denied; a System in an ungranted project is
   indistinguishable from absent.
5. `_CRASH_SIGNATURE` has exactly one definition, reused by both readiness and the watch
   (`first_crash_signature`).
6. The console-truncation-mid-watch edge (`len < mark`) does not crash the watch and cannot
   surface a pre-watch panic.
7. `race-debugging.md` Route 3 points at the tool; the `control` toolset guide and agent index
   list it.
8. Migration `0069` widens `jobs_kind_check`; `test_migrate.py` and a per-migration test stay
   green.
