# M2.8 — Local-libvirt service parity (Integration Contract)

## Purpose

M2.8 closes the gap between what the MCP tool surface **advertises** and what the **default
provider** (`local_libvirt`) actually delivers. Today the build→boot spine is production-wired,
but the live-debug, introspection, and host-dump capture seams are `live_vm`-test-injected
stubs: production `from_env()` wires placeholders that raise `MISSING_DEPENDENCY` ("runs only
under the live_vm gate"), and the real implementations exist only as fixtures the test harness
injects. The tool surface is provider-agnostic (ADR-0063), so an agent walking the catalog
against a default-provider System sails through `runs.build`/`runs.boot` and then hits a wall
of deep, asynchronous `MISSING_DEPENDENCY` failures the moment it tries `debug.start_session`,
`introspect.*`, or `vmcore.fetch` (whose default method, `HOST_DUMP`, local cannot honor).

M2.8 has **two co-equal goals**:

1. **An honest surface for every provider** — the catalog and the per-System view tell the
   truth about which planes a bound provider supports, and the MCP layer **fails fast** with an
   actionable `configuration_error` *before* enqueueing work that cannot succeed, instead of a
   deferred `MISSING_DEPENDENCY` job failure that names a test-harness concept.
2. **Real local-libvirt capability** — production implementations for endpoint resolution
   (gdbstub + drgn-live), offline and live drgn introspection, and host-dump capture, plus the
   completion of the kdump path (#666 / ADR-0207), each **proven live** on a real KVM host.

The honesty goal is **provider-neutral by construction**: the capability descriptor lives on
the universal `ProviderRuntime` seam (ADR-0063), is populated by **every** current provider
(local, remote, fault-inject), and is consumed generically by the MCP layer and
`resources.describe`. A future provider (cloud, bare-metal, PowerVM) inherits honest capability
reporting and fail-fast for free — it only populates the descriptor, never re-plumbs the
surface. This is the mirror of **M2.5** (remote-libvirt capture-method parity): M2.5 brought a
provider up to the surface's promises; M2.8 does the same for the default provider **and**
establishes the cross-provider capability contract that makes the promise checkable.

## What M2.8 adds

- **A provider-neutral capability descriptor on `ProviderRuntime`** (ADR-0208). The existing
  `supported_capture_methods: frozenset[CaptureMethod]` field is generalized into a small set
  of sibling capability frozensets/flags on the universal runtime:
  `supported_debug_transports: frozenset[DebugTransportKind]` (gdbstub, drgn-live) and
  `supported_introspection: frozenset[IntrospectionMode]` (offline-vmcore, live). Every current
  provider's `composition.py` populates them: remote and fault-inject report what they already
  do; local reports build/boot/kdump now and flips debug/introspect/host-dump on **plane by
  plane as Epic B lands**. The descriptor is the *single source of truth* a provider declares
  once and the surface reads everywhere — no provider-kind branching in the MCP layer.
- **Honest per-System capability via `resources.describe`** (ADR-0208). `resources.describe`
  (and the `availability` projection) reports the bound provider's descriptor so an agent can
  ask "what can *this* System actually do?" before it tries — read-only honesty that is correct
  for every provider the moment ADR-0208 lands.
- **Capability-aware MCP admission** (ADR-0209). `debug.start_session`, `introspect.from_vmcore`,
  `introspect.run`, and `vmcore.fetch` consult the **bound** runtime's descriptor and return an
  actionable `configuration_error` *before* enqueueing or running — naming the unsupported
  plane, the provider, and the supported alternative (e.g. "local-libvirt supports KDUMP;
  HOST_DUMP is not available on this provider") — replacing today's deferred, opaque
  `MISSING_DEPENDENCY` job failure.
- **Profile-resolved tool defaults** (ADR-0209). `vmcore.fetch`'s static `HOST_DUMP` default
  (`mcp/tools/lifecycle/vmcore.py`) is removed as a surface that silently steers callers to a
  method the bound provider may not honor. An omitted method is resolved through the existing
  `ProfilePolicy.capture_method(profile)` seam clamped to the core-producing set `{KDUMP, HOST_DUMP}`
  (no new flat default field — that seam already owns the per-System decision); a console/gdbstub
  System with no implicit core method requires an explicit, descriptor-validated argument.
- **Static catalog maturity** (reuses ADR-0175, no new machinery). `debug.*`, `introspect.*`,
  and `vmcore.fetch`'s host-dump path are marked `maturity: "partial"` with a `MaturityReason`,
  a one-line `detail`/`promotion`, and a `providers` pointer ("local-libvirt: planned (M2.8 B*);
  remote-libvirt: implemented"). Pure metadata; the promotion to `"implemented"` happens as the
  matching Epic B plane lands.
- **Production live-debug transport resolution** (ADR-0210). `LocalLibvirtConnect` resolves a
  System's gdbstub and drgn-live endpoints from the **running domain** (the libvirt domain XML
  / per-System config it provisioned), replacing the `_real_resolve_endpoint` /
  `_real_resolve_ssh_endpoint` stubs. `debug.start_session` opens a real loopback transport, and
  the session-bound `debug.*` ops (`set_breakpoint`, `continue`, `read_memory`,
  `read_registers`, `interrupt`, `list_breakpoints`) work against it.
- **Production drgn introspection** (ADR-0210). `LocalLibvirtVmcoreIntrospect` (offline,
  `introspect.from_vmcore`) and `LocalLibvirtLiveIntrospect` (live, `introspect.run`) wire the
  real drgn seams in `from_env()` instead of leaving them `None`, running the fixed in-tree
  helpers (tasks/modules/sysinfo) against a captured core and a live kernel respectively.
- **Production host-dump capture** (ADR-0211). `LocalLibvirtRetrieve`'s host-dump seam
  (`_real_host_dump_capture`) is implemented against the libvirt domain core dump
  (`virDomainCoreDumpWithFlags` / `virsh dump`), giving local a deterministic, guest-cooperation-
  free capture path — the same dual-mode KDUMP+HOST_DUMP shape remote already has (M2.5).
- **Kdump path completion** (#666 / ADR-0207, adopted). The one capture method local *should*
  deliver today is finished and proven: install stages the from-source kernel into the guest
  `/boot/vmlinuz-<ver>` so `kdumpctl` can arm, and a live `control.force_crash` →
  `vmcore.fetch method=kdump` produces a real vmcore.

## Non-goals (scoped out)

- **No new agent-facing MCP tools.** The honest surface is delivered by capability metadata on
  the existing `ProviderRuntime` and existing tools (`debug.*`, `introspect.*`, `vmcore.fetch`,
  `resources.describe`). No new tool is registered.
- **No new debug *features* beyond parity.** Watchpoints, conditional breakpoints, multi-helper
  introspection, and any capability local-libvirt does not already advertise are out of scope —
  M2.8 makes the *advertised* surface real, it does not grow it.
- **No remote-libvirt or fault-inject behavior change.** Those providers only **populate** the
  new descriptor with what they already do (the proof the contract generalizes); their planes
  are untouched.
- **No removal of `local_libvirt`.** M2.8 brings the default provider to parity; its eventual
  removal (deferred since M2) is a separate decision, not this milestone.
- **No multi-host local, no new arch.** Local stays single-host x86_64; ppc64le is M5.
- **No DB/schema change.** The capability descriptor is in-code on `ProviderRuntime`; it is not
  persisted (a provider declares it at composition, not at discovery into `resources.capabilities`).

## Postgres schema (M2.8 delta)

**None.** The capability descriptor is a property a provider declares in code at composition
time, read by the MCP layer and `resources.describe` from the resolved `ProviderRuntime` — it
is not discovered into `resources.capabilities` and needs no column, table, or CHECK change. No
migration this milestone.

## Provider model (M2.8 delta)

`providers/assembly/composition.py` stays the only production assembly point; **no new entry**
— the three current runtimes (`local-libvirt`, `fault-inject`, `remote-libvirt`) each gain the
new descriptor fields in their existing `build_*_runtime()` construction. The change is to the
**shape** of `ProviderRuntime` (ADR-0208 adds the new fields with empty defaults **and flips the
existing `supported_capture_methods` default** from all-methods to empty so every capability field
fails closed — a partial/unconfigured provider reports *no* capability rather than a false
positive; all three providers already set `supported_capture_methods` explicitly, so production
advertising is unchanged), and to the
local provider's `from_env()` seams (Epic B replaces stubs with real implementations). The port
**interfaces are unchanged** — the descriptor is declarative data on the runtime, and the live
seams satisfy the same `Connector` / `VmcoreIntrospector` / `LiveIntrospector` / `Retriever`
ports — so the ADR-0063 portability claim holds and is exercised across all three providers at
once.

```
ResourceKind ──▶ ProviderRuntime (+ capability descriptor: capture methods, debug transports, introspection modes)
  local-libvirt ─▶ build_local_runtime()        # descriptor flips on plane-by-plane as Epic B lands
  fault-inject  ─▶ build_faultinject_runtime()  # descriptor reports its synthetic capability (A1)
  remote-libvirt▶ build_remote_runtime(...)      # descriptor reports its real capability (A1)
```

## MCP tool surface (M2.8 delta)

No new tools. Three changes to **existing** tools, all provider-neutral:

1. **Capability-aware admission** — `debug.start_session`, `introspect.from_vmcore`,
   `introspect.run`, `vmcore.fetch` reject an unsupported plane/method against the bound
   runtime's descriptor with `configuration_error` (data: the unsupported capability, the
   provider, the supported set) **before** enqueue/execution (ADR-0209).
2. **Profile-resolved `vmcore.fetch` default** — the static `HOST_DUMP` default is dropped; an
   omitted method resolves through the existing `capture_method(profile)` seam clamped to the
   core-producing set, else a required descriptor-validated argument (ADR-0209).
3. **Maturity metadata** — `debug.*`, `introspect.*`, and the host-dump path carry
   `maturity: "partial"` + provider pointer until the matching Epic B plane promotes them to
   `"implemented"` (ADR-0175 mechanism, no new field). `resources.describe` projects the
   descriptor (ADR-0208).

The `test_tool_docs` drift guard and the generated `docs/guide/reference/*` regenerate for the
maturity/`describe` changes.

## Auth / RBAC delta

**None.** Capability reporting and admission run inside the existing per-project
`operator`/`viewer` checks for each tool; no new role, claim, or gate. `resources.describe`
keeps its current viewer-visibility rules; the descriptor carries no tenant-sensitive data
(it is provider-kind capability, not host detail).

## Error taxonomy (M2.8 delta)

**None.** Capability rejections use the existing `configuration_error`
(`CONFIGURATION_ERROR`) — enriched with `data.reason` / supported-set detail per ADR-0166's
actionable-detail pattern, not a new category. The Epic B live seams continue to map real
failures to existing categories (`transport_failure`, `debug_attach_failure`,
`readiness_failure`, `infrastructure_failure`, `missing_dependency` only for a genuinely absent
host dependency such as an unimported `drgn`/`guestfs` — never as a stand-in for an unwired
seam). The rule is unchanged: most specific existing value, no new strings.

### Sequencing & shared seams (no separate plan)

M2.8 follows the M1.3/M1.4/M1.5/M2 model: detailed design lives here and in the four ADRs;
each issue is planned and implemented end-to-end by `work-issue` (the per-issue spec is authored
alongside its ADR during the issue, per the project's spec-with-ADR convention). There is **no
`docs/plans/m2.8-implementation.md`**. The cross-issue concerns no single issue owns:

- **Descriptor first, then enforcement, then planes.** A1 lands the descriptor (declarative,
  read-only-honest via `describe`); A2 lands the enforcement; only then do the Epic B planes
  fill it in. A plane PR (B*) both wires the real seam **and** flips its descriptor field +
  promotes its maturity in the same commit, so the surface is never wired-but-still-marked-partial
  or marked-implemented-but-stubbed.
- **Live verification is per-issue, on the development KVM host.** Each Epic B issue closes only
  after a live drive on this host proves the plane (the toolchain — KVM, libvirt `qemu:///session`,
  gdb, drgn, libguestfs, virt-builder — is present). The `live_vm`-marked tests for the plane
  are run locally (they stay skipped in hosted CI, which has no KVM); the hermetic fake-seam
  unit/contract tests gate the PR in CI. B6 is the milestone-end end-to-end proving run.
- **Expected rebase zones:** `providers/core/runtime.py` (A1 adds descriptor fields; every B*
  flips one), `providers/assembly/composition.py` and the three `composition.py` providers (A1
  populates the descriptor in each), the generated `docs/guide/reference/*` and `test_tool_docs`
  (A3 + each maturity promotion — regenerated, not hand-edited), and `mcp/tools/lifecycle/vmcore.py`
  (A2 default change; B4/B5 capture methods).
- **Each issue runs in its own worktree, outside the repo tree** (parallel `work-issue` requires
  it, per the global standard).

## Carried invariants

1. **The provider seam is unchanged in kind** (ADR-0063) — the capability descriptor is
   declarative data added to the `ProviderRuntime` dataclass, and the live seams satisfy the
   same `Connector`/`VmcoreIntrospector`/`LiveIntrospector`/`Retriever` ports; M2.8 changes the
   runtime's shape (new fields) and the local provider's seam *implementations*, not the port
   interfaces — so the portability claim holds and is now exercised across all three providers.
2. **The surface never lies, at any merge point** — A gates B, and each B plane flips its
   descriptor + promotes its maturity in the same PR that wires it. A partial state reports
   *less* capability (conservative defaults), never more; an agent is never told a stubbed plane
   works.
3. **`MISSING_DEPENDENCY` means a genuinely absent host dependency, not an unwired seam** — after
   M2.8 the live seams are real; `MISSING_DEPENDENCY` is reserved for the operator-provided
   dependency truly missing at runtime (e.g. `drgn`/`guestfs` not importable), surfaced up front,
   never as a stand-in for "this provider never implemented this plane."
4. **Capability honesty is provider-neutral and future-proof** (ADR-0208) — the descriptor lives
   on the universal runtime, every current provider populates it, and the MCP layer/`describe`
   consume it without provider-kind branching; cloud/bare-metal/PowerVM inherit honest reporting
   and fail-fast by populating the descriptor, never by re-plumbing the surface.
5. **Live capability is proven on hardware, not asserted** — each Epic B plane is driven live on
   a real KVM host before its maturity promotes to `"implemented"`; CI covers the seam contract
   with fakes, the development host proves the real arm/attach/capture, and B6 records the
   milestone-end end-to-end run.
