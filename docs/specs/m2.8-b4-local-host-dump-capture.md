# Spec — M2.8 B4: local-libvirt host-dump capture via libvirt domain core dump

- **Issue:** [#678](https://github.com/randomparity/kdive/issues/678) (Epic B / M2.8)
- **ADR:** [ADR-0211](../adr/0211-local-libvirt-host-dump-capture.md) (anchor; do not re-decide)
- **Status:** Draft → implemented on `feat/local-host-dump-678`
- **Depends on:** #672 (A1, narrowed local `supported_capture_methods` to `{KDUMP}`),
  #674 (A2), #679 (B5, kdump live proof — gates the tool-maturity promotion, deferred here).

## Problem

`LocalLibvirtRetrieve.capture` branches to `_capture_host_dump` for `CaptureMethod.HOST_DUMP`,
but the injected `host_dump_capture` seam's real implementation, `_real_host_dump_capture`,
raises `MISSING_DEPENDENCY` unconditionally. So a `vmcore.fetch` call that resolves HOST_DUMP on
a local System fails, and KDUMP (the ADR-0203 overlay harvest) is local's only real capture path.
Local also no longer advertises HOST_DUMP after A1 narrowed `supported_capture_methods` to
`{KDUMP}`.

## Decision (from ADR-0211, restated for scope)

Implement `_real_host_dump_capture` against the **libvirt domain core dump**
(`virDomainCoreDumpWithFlags` / `domain.coreDumpWithFormat`), writing the running/crashed
domain's memory image to a private, worker-readable temp path, then funnelling that path through
the **same file-streaming store discipline KDUMP uses** (#657): raw `sensitive` core streamed to
the object store, redacted dmesg derivative, build-id read from the core file, ADR-0050
method-encoded key `vmcore-host_dump`. Re-advertise `HOST_DUMP` in local's
`supported_capture_methods`.

## In scope

1. **Refactor `_capture_host_dump` to the file-streaming path.** ADR-0211 §3 and its rejected
   alternative ("Hold the whole core in memory") are explicit: host-dump reuses the KDUMP
   temp-file streaming discipline, not the legacy in-memory `bytes` buffer. The
   `host_dump_capture` seam type changes from `Callable[[UUID], bytes | None]` to
   `Callable[[UUID], Path | None]`. KDUMP and HOST_DUMP then share one private
   `_capture_via_file(system_id, method, core)` helper that streams the raw core, persists the
   redacted derivative from the file, reads the build-id from the file, and removes the spool dir
   in a `finally`.

2. **Implement the real seam.** `_real_host_dump_capture(system_id) -> Path | None`
   (`# pragma: no cover - live_vm`):
   - Open the provider's configured libvirt URI (`_libvirt_uri()`, `KDIVE_LIBVIRT_URI`, default
     `qemu:///system`) — never `libvirt.open(None)`, matching `control.py`/`discovery.py`.
   - Look up the domain by `domain_name_for(system_id)`. A missing domain returns `None` (no
     core to capture → the orchestrator surfaces `READINESS_FAILURE` via `_no_core`, identical to
     the kdump no-core path).
   - **Require a live domain.** `coreDumpWithFormat` can only dump an *active* domain's memory.
     Unlike kdump (which `_force_off_domain`-destroys the domain to read its overlay offline),
     host-dump dumps the **live** domain in place and never force-offs it — matching remote, which
     dumps the active domain directly. `vmcore.fetch` admits on a `CRASHED` System, but a crashed
     guest may already be inactive (the local domain XML sets no explicit `on_crash`, so libvirt's
     default policy applies and a panicked/halted guest can be shut off). So before dumping, check
     `domain.isActive()`; an inactive domain returns `None` → `READINESS_FAILURE` ("no live domain
     to dump"), the same actionable category as no-core, **not** a generic
     `INFRASTRUCTURE_FAILURE`. (A residual `coreDumpWithFormat` libvirt error on an active domain
     still maps to `INFRASTRUCTURE_FAILURE`.)
   - Create a private `mkdtemp(prefix="kdive-host-dump-")` spool dir and dump to `spool/vmcore`
     via `domain.coreDumpWithFormat(path, VIR_DOMAIN_CORE_DUMP_FORMAT_RAW, VIR_DUMP_MEMORY_ONLY)`
     (mirrors remote's format/flags: a crash-format, memory-only core the downstream `crash`/drgn
     postmortem reads).
   - Enforce the `MAX_CORE_BYTES` (5 GiB) ceiling on the produced file; over-ceiling is
     `CONFIGURATION_ERROR` and the spool is removed. **The ceiling is enforced post-write on
     local** (the core is already on the worker temp dir): unlike remote, which can read the dump
     volume's capacity *before* a stream download, the local domain dumps straight to the worker
     filesystem, so there is no pre-write size to inspect. This is acceptable — the dump target is
     a private worker temp dir removed on rejection, and a local domain's RAM is bounded by its
     System shape; pre-flighting against the domain's configured memory is out of scope. The
     asymmetry with remote's pre-download capacity check is intentional.
   - Return the spool `Path`; the caller (`capture`) owns cleanup on the returned path, exactly as
     `_real_wait_for_vmcore` does for kdump. On any failure before a successful return the spool
     dir is removed here.

3. **Re-advertise HOST_DUMP.** Add `CaptureMethod.HOST_DUMP` to local's
   `supported_capture_methods` in `composition.py` (the dual-mode `{KDUMP, HOST_DUMP}` shape
   remote already has).

4. **Remove the now-dead bytes seams.** `extract_redacted` (the bytes dmesg seam) and its stub
   `_real_extract_redacted_bytes` were only used by the in-memory host-dump path; after the
   refactor nothing calls them, so they are deleted (CLAUDE.md "Replace, don't deprecate" / no
   dead code). The bytes `read_vmcore_build_id` seam stays — `run_crash_postmortem` still uses it
   to read the build-id from object-store bytes.

5. **vmcore.fetch tool-maturity: keep `partial`.** Per the orchestrator's M2.8 sequencing, the
   tool maturity does **not** promote to `implemented` in this PR. The host-dump path is wired but
   not yet hardware-proven on KVM in this environment (design-doc invariant 5; the live
   `virsh dump` → store → crash/drgn round-trip is B6 / #680, run post-merge). The `vmcore.fetch`
   `providers` meta string is updated to state local host-dump is **wired, pending live KVM proof
   (M2.8 B6 #680)** rather than "planned (M2.8 B4)". A tiny follow-up promotes maturity after the
   live drive.

## Out of scope

- The actual live KVM drive / `crash`/drgn round-trip (B6 / #680).
- Promoting `vmcore.fetch` maturity to `implemented` (deferred, see §5).
- Any storage-pool/volume machinery — unlike remote, the local domain runs on the worker host, so
  the core dumps straight to a host-local path; no `storageVolLookupByName`, `pool.refresh`, or
  libvirt stream download is needed.
- KDUMP behaviour, ports, schema, migrations (all unchanged).

## Acceptance criteria

**CI (fakes, non-gated):**

- `capture(sys, HOST_DUMP)` with a fake `host_dump_capture` returning a spooled `Path` streams the
  raw core via `put_stream` under key `…/vmcore-host_dump`, persists `…/vmcore-host_dump-redacted`
  from the **file** dmesg seam, returns the file build-id and `raw_size_bytes == core.stat().st_size`.
- The HOST_DUMP path removes the spool dir on success **and** on store failure (mirrors the
  existing kdump spool-cleanup tests).
- `host_dump_capture` returning `None` raises `READINESS_FAILURE`.
- The stored-checksum verification (`_verify_stored`) applies to the HOST_DUMP raw put, same as
  kdump (over-the-wire corruption is `INFRASTRUCTURE_FAILURE`).
- A store failure on the HOST_DUMP raw put is `INFRASTRUCTURE_FAILURE` and still removes the spool.
- `composition.build_runtime(...).supported_capture_methods == {KDUMP, HOST_DUMP}`.
- The bytes `extract_redacted` seam is gone (constructor no longer accepts it); no test or call
  site references it.
- Generated tool-reference docs + `test_tool_docs` stay green with `vmcore.fetch` at `partial` and
  the updated `providers` string.

**Live (KVM host, gated `live_vm`, NOT run in CI — orchestrator runs post-merge for B6):**

- A real `virsh dump` of a local System's domain produces a core stored under `vmcore-host_dump`,
  whose build-id and dmesg are read back and a `crash`/drgn postmortem round-trips.

## Failure-mode table

| Condition | Category | Where |
|---|---|---|
| Domain absent / not found | `READINESS_FAILURE` (via `None` → `_no_core`) | `_real_host_dump_capture` → `capture` |
| Domain present but inactive (crashed-then-shutoff) | `READINESS_FAILURE` (via `None` → `_no_core`) | `_real_host_dump_capture` `isActive()` guard |
| Core exceeds 5 GiB (post-write) | `CONFIGURATION_ERROR`; spool removed | `_real_host_dump_capture` ceiling check |
| `coreDumpWithFormat` libvirt error (active domain) | `INFRASTRUCTURE_FAILURE` | `_real_host_dump_capture` |
| drgn (build-id/dmesg) absent | `MISSING_DEPENDENCY` (surfaced, not degraded) | shared core-file seams |
| dmesg extraction non-missing-dep failure | degrade to `DMESG_UNAVAILABLE` | `redact_dmesg` (shared) |
| Object-store put failure | `INFRASTRUCTURE_FAILURE`; spool removed | `_capture_via_file` |
| Stored checksum mismatch | `INFRASTRUCTURE_FAILURE` | `_verify_stored` |

## Notes / invariants preserved

- ADR-0050 first-method-wins per System: HOST_DUMP and KDUMP store under distinct method-encoded
  keys; `vmcore.fetch`'s same-method-dedup / different-method-conflict checks are unchanged.
- No whole-core in-memory buffer on either path (#657).
- The real libvirt core-dump call is the only new `# pragma: no cover - live_vm` line; all
  orchestration (branch, stream, redact, build-id, ceiling, cleanup) is fake-driven in CI.
